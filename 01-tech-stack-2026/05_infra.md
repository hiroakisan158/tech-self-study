# 05. インフラとデータ — 「とりあえず Postgres」と、デプロイの宗派

インフラは One-way door の宝庫(特に DB)であると同時に、
**振り子がもっとも大きく揺れてきた**レイヤーです。オンプレ → クラウド →
サーバーレス/エッジ → そして 2026 年、「セルフホスト回帰」の揺り戻しが来ています。

---

## 1. データベース — 珍しく大同団結が成立した場所

### 「とりあえず Postgres(Just use Postgres)」

2026 年のデータベース選定は、ほぼ 1 行で終わります:
**明確な理由がない限り PostgreSQL**。

NoSQL ブーム(2010 年代)の反省を経て、Postgres は拡張によって
「なんでも屋」になりました:

| 「専用 DB が要る」と言われていたもの | Postgres での答え |
|---|---|
| ドキュメント DB(MongoDB) | JSONB |
| 全文検索(Elasticsearch) | 組み込み全文検索(小〜中規模なら) |
| ベクトル検索(Pinecone 等) | **pgvector**(AI ブームで一気に標準化) |
| ジョブキュー(Redis + Sidekiq 等) | `SELECT ... FOR UPDATE SKIP LOCKED` / pgmq |
| 地理空間 | PostGIS |

> 💡 **ポイント**: これは「専用 DB が無意味」ではなく、**「専用 DB は規模が
> 要求してから足す」**という順序の話です。[01 章](01_principles.md)の
> イノベーション・トークンを DB に使うな、とも言い換えられます。

### 残る流派

- **サーバーレス Postgres 派** — Neon(2025 年に Databricks が買収)、Supabase
  (BaaS として一大勢力)、PlanetScale(MySQL の雄が Postgres 対応に参入)。
  「スケールゼロ・従量課金・ブランチ機能」で Vercel 型スタックと好相性
- **SQLite 復権派** — 「1 台のサーバー + SQLite + レプリケーション(Litestream)で、
  世の中の大半のサービスは足りる」という過激派が 2020 年代に台頭。
  シンプルさの極北として、セルフホスト派と思想的に連合
- **MySQL** — 新規採用は減ったが、既存資産と運用ノウハウで健在
- キャッシュは Redis のライセンス変更(2024)を機に **Valkey**(OSS フォーク)へ
  移行する動きが定着。あるいは「キャッシュも Postgres/アプリ内でよくない?」派も

## 2. デプロイ — 2026 年最大の宗教戦争のひとつ

[02 章](02_frontend.md)の Next.js vs TanStack 論争の裏側には、常にこの戦線があります。

### 四大宗派

| 宗派 | 代表 | 思想 | 典型的な破綻 |
|---|---|---|---|
| **フロント特化 PaaS** | Vercel、Netlify、Cloudflare | git push だけ。プレビュー環境・CDN・画像最適化込み | スケール時の請求額(「Vercel 税」)、WebSocket 等の常駐処理 |
| **コンテナ PaaS** | Railway、Fly.io、Render、Cloud Run | 常駐プロセス+DB を丸ごと。コールドスタートなし | 完全マネージドほどは楽でない |
| **セルフホスト回帰** | VPS(Hetzner 等)+ **Coolify** / Dokploy / Kamal | 月数十ドルで全部動く。ロックインなし | **運用は自分**。セキュリティパッチも自分 |
| **ハイパースケーラー直** | AWS / GCP / Azure | 何でもできる。エンタープライズの既定 | 複雑性と専任者コスト |

### セルフホスト回帰は何の揺り戻しか

2024〜26 年に「Vercel から VPS に移して月額 1/10 になった」系の記事が量産されました。
DHH(Rails 作者)の「クラウド脱出」運動、Coolify のような
「自分の VPS を Vercel 風に使う」OSS の成熟が追い風です。ただし
[01 章](01_principles.md)の前提分析を忘れずに:

- **PaaS 派の前提**: エンジニアの時給 > サーバー代。運用専任がいない。
  プレビュー環境が開発フローの核
- **セルフホスト派の前提**: トラフィックが読める。運用できる人がいる(そして
  その人が辞めない)。ロックイン回避に価値を置く

> ⚙️ **Next.js のセルフホスト**は v15 以降大きく改善し(Docker standalone 出力、
> キャッシュ制御の改善)、OpenNext(AWS / Cloudflare 向け変換層)も成熟しました。
> 「Next.js = Vercel 専用」は 2026 年にはもう正確ではありません。ただし
> ISR・画像最適化などを完全再現する運用コストは残ります([06 章](06_playbook.md))。

### エッジコンピューティングの現在地

「全部エッジで動かす」という 2022〜23 年の熱狂は冷めました。
**データ(DB)が 1 リージョンにある限り、計算だけ端に置いても速くならない**
という当たり前の物理が再確認され、2026 年の整理は:

- **エッジ向き**: ルーティング、認証チェック、A/B 振り分け、静的配信(= CDN の延長)
- **リージョン向き**: DB を触る処理ほぼすべて
- 例外として **Cloudflare Workers 経済圏**(D1、Durable Objects、R2)は
  「データもエッジに置く」独自路線で別惑星として成立中

## 3. コンテナとオーケストレーション

- **Docker(コンテナ化)は無風のデファクト**。Go の単一バイナリは
  scratch イメージで数 MB になり、コンテナと相性最高([go-fable-101](../03-go-fable-101/README.md))
- **Kubernetes** は「使う必要が出るまで使うな」が 2026 年の共通見解。
  マルチサービス・マルチチームの規模になったら本領。それまでは PaaS か
  Compose で十分
- マイクロサービス論争は「**モジュラーモノリスから始め ->組織が分割を要求したら切る**」
  で実質決着(Conway の法則の再発見)

## 4. IaC・CI・可観測性(手短に)

- **IaC**: Terraform のライセンス変更(2023)で **OpenTofu**(OSS フォーク)が分立。
  機能はほぼ同等で、思想(オープンさ)で選ぶ状態。TS で書きたい派は Pulumi / SST
- **CI**: **GitHub Actions が無風のデファクト**。高速化したければ self-hosted runner
  か Depot 等のアクセラレータ
- **可観測性**: 計装の標準は **OpenTelemetry で決着**(ベンダーロックインは
  バックエンド側だけの問題になった)。バックエンドは Datadog(高機能・高額)vs
  Grafana スタック / 新興(Axiom、Honeycomb 等)。Go では構造化ログが
  stdlib(`log/slog`)入りしたのが象徴的
- **エラートラッキング**: Sentry がデファクト

## 5. この章の選定サマリー

| 決めること | 迷ったらこれ | それを覆す条件 |
|---|---|---|
| DB | PostgreSQL(マネージド) | 単一サーバー・小規模 → SQLite も真剣な選択肢 |
| キャッシュ | まず Postgres で粘る → 必要なら Valkey | — |
| フロントのデプロイ | Vercel(小規模は Hobby/Pro で十分) | 請求額が痛くなったら OpenNext / コンテナ移行を検討 |
| バックエンド(Go)のデプロイ | コンテナ PaaS(Cloud Run / Fly / Railway) | 社内標準が AWS → ECS/Fargate |
| K8s | 使わない | マルチチーム・マルチサービスになったら |
| IaC | Terraform / OpenTofu | TS 全振りチーム → SST/Pulumi |
| 可観測性 | OpenTelemetry + Sentry + (Grafana or Datadog) | — |
