# 03. バックエンド — 言語の選び方、API 境界、そして ORM 戦争

バックエンドはフロントエンドより変化が遅く、そのぶん **One-way door
(不可逆な決定)が集中するレイヤー**です([01 章](01_principles.md))。
言語・DB・公開 API のスキーマは、あとから変えるコストが桁違いに高い。

---

## 1. 言語選定 — 「フルスタック TS」対「適材適所ポリグロット」

2026 年のバックエンド言語論争の最大の対立軸はこれです。

**フルスタック TypeScript 派**
- 言語 1 つでフロントからバックまで。型・バリデーション・ユーティリティを共有できる
- 少人数で最速。個人開発〜アーリースタートアップの支配的選択
- 弱点: CPU 重い処理・大量並行接続は [JS ランタイムの構造的弱点](../04-typescript-fable-101/language-overview/README.md)

**適材適所ポリグロット派**
- API サーバー・インフラツールは Go、AI/ML・データ処理は Python、UI は TS
- 各言語の主戦場に素直。中規模以降の組織の支配的選択
- 弱点: 境界(API スキーマ)の管理コストと、人材の分断

| 言語 | 2026 年の主戦場 | 選ぶ理由の典型 |
|---|---|---|
| **TypeScript** | BFF、フルスタック app、Serverless | フロントと型を共有したい/チームが小さい |
| **Go** | API サーバー、インフラ・CLI、高並行処理 | 運用のシンプルさ(単一バイナリ)、goroutine、チームの規律 |
| **Python** | AI/ML、データ基盤、社内自動化 | エコシステムがそこにしかない([overview](../02-python-fable-101/language-overview/README.md)) |
| **Rust** | 性能クリティカル、**他言語のツール基盤** | JS ツールチェーンを裏で支配中([04 章](04_toolchain.md)) |
| Java / Kotlin / C# | 大企業基幹、金融、Android | 既存資産と組織の標準 |

> 💡 **あなたの転職先(Go + Next.js)はポリグロット派の王道構成**です。
> 「UI とフロント配信は Next.js、ビジネスロジックと API は Go」という分業は、
> 2026 年の中規模 Web 企業でもっとも標準的な布陣のひとつ。この構成の設計論点は
> [06 章](06_playbook.md)で専用に扱います。

## 2. Go のバックエンド — 「stdlib ファースト」という文化

Go 界隈の技術選定は JS 界隈と文化が根本的に違います。合言葉は
**「まず標準ライブラリ、足りない分だけ薄く足す」**。

### HTTP ルーティング

- **Go 1.22 で標準の `http.ServeMux` がメソッド・パスパラメータ対応**になり、
  「小さな API ならフレームワーク不要」が現実になった(この影響で、かつての王者
  gorilla/mux は 2023 年に一度アーカイブされる事件も)
- **chi** — stdlib 完全互換の薄いルーター。「stdlib ファースト派」の実質標準
- **Gin** — 利用率首位(JetBrains 調査で約 48%)。エコシステムと情報量で選ばれる。
  独自 Context 型が stdlib から少し離れるのが好みの分かれ目
- **Echo** — Gin と同ポジションでやや行儀がよい API
- **Fiber** — fasthttp ベースで高速だが **net/http 互換を捨てている**(HTTP/2 制約など)。
  Go 文化圏では「速度のために標準を捨てる」少数派

> ⚖️ **対立の構図**: JS の「フレームワークが土台」に対し、Go は「stdlib が土台、
> フレームワークは装飾」。Go 選定で Rails/Laravel 型のフルスタックフレームワークを
> 探し始めたら文化の読み違いです([go-fable-101 overview](../03-go-fable-101/language-overview/README.md))。
> なお Go チームは 2025 年、長年のエラーハンドリング構文論争に「**新構文は追加しない**」
> と公式に幕を引きました。`if err != nil` は仕様であり文化です。

### DB アクセス(Go 版 ORM 論争)

| 流派 | 代表 | 思想 |
|---|---|---|
| **SQL 回帰派(優勢)** | **pgx + sqlc** | SQL を書き、**コード生成で型を得る**。「SQL は隠すな、型は諦めるな」 |
| ORM 派 | GORM、ent | 生産性優先。GORM は「暗黙の挙動が多い」と批判も根強い。ent(Meta 発)はスキーマ・アズ・コード派 |
| 中間派 | sqlx | stdlib + 構造体マッピングだけ |

2026 年の新規プロジェクトでは **sqlc が推されることが目立って増えました**。
「SQL ファイルからタイプセーフな Go コードを生成する」方式は、後述の
OpenAPI コード生成と同じ「**スキーマ駆動**」思想の DB 版です。

## 3. TypeScript のバックエンド

| フレームワーク | 位置づけ |
|---|---|
| **Hono** | 2026 年の主役。超軽量・Web 標準 API 準拠・**マルチランタイム**(Node/Bun/Deno/Cloudflare Workers 全対応)。新規の小〜中規模 API の第一候補 |
| **Fastify** | Node 界の堅実な高速枠。プラグイン設計が成熟 |
| **Express**(v5) | 依然として最大シェアだが「レガシーの王」。新規で積極採用する理由は薄い |
| **NestJS** | エンタープライズ派(DI・デコレータ・Angular 風の規律)。大規模チームで規律を買う選択 |
| Elysia | Bun 専用の最速枠。Bun に賭けるなら |

Hono の躍進は単なる速度ではなく、**「Web 標準 API(Request/Response)だけに依存し、
どのランタイムでも動く」**という設計が、ランタイム群雄割拠([04 章](04_toolchain.md))
の時代にヘッジとして機能しているからです。

## 4. API 境界 — フロントとバックの「国境」をどう引くか

ポリグロット構成(Go + Next.js)で最重要の設計判断です。2026 年の勢力図:

| 方式 | 状態 | 向くケース |
|---|---|---|
| **REST + OpenAPI(スキーマ駆動)** | **デファクト**。スキーマから TS 型/クライアント/Go サーバースタブを自動生成 | 言語をまたぐ境界すべて。**Go + Next.js 構成の標準解** |
| **tRPC** | ピークを過ぎた。TS モノレポ**限定**なら今も快適 | フロント・バック両方 TS で同一リポジトリのとき |
| **GraphQL** | 2 度目の幻滅期。新規採用は減少、大組織の federation に残存 | 多数のクライアントが同一データを別形状で欲しがる大組織 |
| **gRPC / Connect** | サービス**間**通信のデファクト | マイクロサービス内部。ブラウザ向けには Connect で緩和 |
| **Server Actions**(Next.js) | Next 内部の準標準 | Next アプリ内で完結するミューテーション |

> 📜 **GraphQL 退潮の教訓**: 「クライアントが欲しい形でデータを取れる」という理想は
> 本物でしたが、キャッシュ・N+1・認可・複雑性のコストが、**複数チーム・複数クライアント
> という前提を持たない組織**には見合いませんでした。「Facebook の問題を輸入するな」
> ([01 章](01_principles.md)チェックリスト 2 番)の代表例として記憶されています。
> tRPC も同様に「TS モノレポ」という前提の外(まさに Go バックエンド構成)では使えない
> ことが冷静に認識され、**汎用解はスキーマ駆動 REST に回帰**しました。

### バリデーション — 「境界では必ず実行時検証」が常識に

[TS の型は実行時に消える](../04-typescript-fable-101/language-overview/README.md)ため、
外部入力の検証ライブラリは必須装備になりました。

- **Zod(v4)** — デファクト。「スキーマから型を導出」のパラダイムを確立
- **Valibot**(バンドルサイズ極小)、**ArkType**(実行速度)が対抗
- 2025 年に **Standard Schema**(共通インターフェース)が普及し、
  ライブラリ間の互換問題はほぼ解消——**どれを選んでも詰まない**時代に

## 5. TS の ORM 戦争 — Drizzle vs Prisma

2026 年のバックエンド TS でいちばん熱い戦線です。**2026 年 5 月、npm 週間 DL 数で
Drizzle が Prisma を初めて逆転**しました(累計利用では Prisma が依然首位)。

| | **Prisma** | **Drizzle** |
|---|---|---|
| 思想 | スキーマファースト(独自 `.prisma` 言語)。**SQL を隠す** | コードファースト(スキーマも TS)。**SQL に寄り添う** |
| 強み | DX・ドキュメント・マイグレーション体験・実績 | 軽量(サーバーレス/エッジで圧勝)・SQL 知識がそのまま活きる・依存ゼロ |
| 弱み | 重量級。複雑クエリで生 SQL に逃げがち | マイグレーション周りが若い。抽象度が低いぶん書く量は増える |
| 転機 | Rust 製エンジンを捨てて TS へ書き換え(軽量化で Drizzle を追撃中) | Hono・Astro など新世代エコシステムの既定に採用され続ける |

これは[01 章](01_principles.md)の「抽象化派 vs SQL 回帰派」対立の TS 版です。
そして業界全体のトレンドは明確に **SQL 回帰**(Go の sqlc、TS の Drizzle/Kysely)。
「SQL は隠すべき実装詳細ではなく、キャリアを通じて償却できる資産」という再評価が
2020 年代後半の通奏低音です。第 3 極として「ORM は要らない、型付きクエリビルダで十分」
という **Kysely 派**もいます。

## 6. 認証 — 「自前で書くな」から「どこまで外に出すか」へ

| 流派 | 代表 | 向くケース |
|---|---|---|
| **ライブラリ自前ホスト派** | **Better Auth**(TS、急成長中) | 標準解。データを手元に置きつつ実装は借りる |
| SaaS 派 | Clerk、Auth0 | 速度優先の MVP、認証を差別化と見なさない組織 |
| 自前 OSS 基盤派 | Keycloak、Ory | 大組織・コンプライアンス要件・マルチプロダクト SSO |
| Go 自前実装派 | stdlib + ライブラリ少々 | Go 文化では session/JWT 自前実装が今も普通 |

長年の定番だった Auth.js(旧 NextAuth)は開発が停滞し、コミュニティの新規流入は
Better Auth に移りました。なお 2025 年の Next.js Middleware 脆弱性
(CVE-2025-29927)以降、**「認可はフレームワークの入口でなくデータ層の近くで行う」**
という多層防御の原則が改めて共有されています。

| 決めること | 迷ったらこれ | それを覆す条件 |
|---|---|---|
| バックエンド言語 | チーム既存スキル > 理想論。新規なら TS(小規模)/ Go(運用重視) | ML・データ → Python 一択 |
| Go の HTTP | stdlib + chi | 情報量・既存資産重視 → Gin |

| TS ORM | 新規は Drizzle、実績重視は Prisma | SQL 好きのチーム → Kysely や生 SQL も正解 |
| 認証 | Better Auth(TS)/ 自前 + ライブラリ(Go) | 急ぐ MVP → Clerk |
