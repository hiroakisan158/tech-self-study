# 04. ツールチェーン — JS の道具は Rust(と Go)で作り直されている

2020 年代後半のツールチェーンを貫く一本の物語があります:
**「JS で書かれた JS の道具が、ネイティブ言語製に総入れ替えされつつある」**。
バンドラは Rust 製(Turbopack、Rspack、Rolldown)、リンタも Rust 製(Biome、oxlint)、
そして **TypeScript コンパイラ自体が Go 製(tsgo)に**——
あなたが学んでいる Go が、TS の足元を支える時代です。

---

## 1. TypeScript コンパイラ — tsgo という事件

2025 年、Microsoft は TS コンパイラを **Go に移植**して約 10 倍高速化する計画
(通称 tsgo、製品名 TypeScript 7)を発表し、2026 年 4 月にパブリックベータが
出ました。エディタの型チェックが体感即時になるインパクトで、
2026 年後半〜2027 年の本番投入が視野です。

> 📜 **なぜ Rust でなく Go だったのか**: 既存 tsc のコードベース(GC 前提・
> 循環参照だらけの AST)を**そのまま移植**するには、GC を持ち構造が近い Go が
> 最適だった——という Microsoft の説明は、[Go の設計思想](../go-fable-101/language-overview/README.md)
> (実用主義・移行コスト重視)の好例としてよく引かれます。
> 「性能が欲しい = Rust」ではない、という技術選定の教材みたいな話です。

---

## 2. ランタイム — Node の安定 vs Bun の速度

| ランタイム | 2026 の位置づけ |
|---|---|
| **Node.js**(LTS) | 本番のデファクト。退屈で、それが最大の長所。npm 互換性 100% という絶対的強み |
| **Bun**(1.3 世代) | ランタイム+パッケージマネージャ+テスト+バンドラの all-in-one。**開発ツールとしては既に主流級**、本番ランタイムとしても採用が広がる |
| **Deno**(v2) | npm 互換化で実用性は上がったが、シェアは伸び悩み。セキュリティモデルは今も独自の強み |

現実的な運用は「**開発は Bun で高速に、本番は保守的に Node**」というハイブリッドが
多数派です。ベンチマークの「Bun は 2〜4 倍速い」は素の HTTP の話で、
**DB アクセスが入ると差はほぼ消える**ことも冷静に共有されています。
なお [Hono](03_backend.md) のような Web 標準準拠フレームワークを選んでおくと、
ランタイム選択が Two-way door(可逆)になります——これが 2026 年的なヘッジです。

## 3. パッケージマネージャ — pnpm がデファクト

- **pnpm** — デファクト。速く、ディスク効率がよく、モノレポ(workspace)標準。
  主要スキャフォルダの既定になった
- **bun install** — 純粋な速度なら最速。Bun 採用チームはこれ一本化
- **npm** — 無難枠として健在。**yarn** は退潮
- モノレポのタスク実行は **Turborepo**(pnpm workspace + turbo が軽量標準)、
  高機能が必要なら **Nx**

## 4. リンタ/フォーマッタ — 三国志の真っ最中

2026 年でもっとも「どれにすべき?」と聞かれる戦線です。

| 陣営 | 構成 | 強み | 弱み |
|---|---|---|---|
| **既存王朝** | ESLint + Prettier | プラグイン資産が圧倒的(React Hooks、a11y、社内ルール)。型情報リント完備 | 遅い。設定が複雑。ツール 2 つの調停が面倒 |
| **統一新興** | **Biome**(v2) | 1 ツールで lint+format、10〜25 倍速。v2 で **tsc 不要の型認識リント**という技術的飛躍 | プラグイン資産の再現がまだ部分的 |
| **VoidZero 帝国** | oxlint + oxfmt | 最速(ESLint 比 50〜100 倍)。Vite/Rolldown と同じ oxc 基盤で垂直統合 | 最後発。エコシステム形成中 |

選定の現実解:
- **既存の大規模リポジトリ** → ESLint + Prettier 続投(移行の益がコストに見合うまで待つ)
- **新規プロジェクト** → Biome が有力。設定ほぼゼロで速い
- **Vite 中心のチーム** → oxlint は「ESLint と併走」から始められるので様子見しやすい

> ⚖️ **対立の裏にある構図**: VoidZero は Vue/Vite の Evan You が VC 出資で設立した
> 企業で、Vite・Rolldown・oxc・Vitest を垂直統合しています。「JS 基盤ツールが
> 一企業に集中する」ことへの期待(持続可能な開発資金)と警戒(方針転換リスク)は、
> [01 章](01_principles.md)の「誰が金を出しているか」で読むべき典型例です。

## 5. ビルド/バンドラ — Vite の一強と、その足元の交代劇

- **Vite** — SPA・ライブラリ・フレームワーク基盤(TanStack Start、Astro、SvelteKit…)
  として一強。もはや「ビルドツール」でなく **エコシステムの共通土台**
- **Rolldown** — Vite の内部(esbuild + Rollup の 2 本立て)を単一の Rust 製に
  置き換える計画。Vite 8 世代で移行が進行中(2026 年前半時点ベータ)。
  ユーザーからは「Vite が速くなるだけ」に見えるのが理想形
- **Turbopack** — Next.js 専用(v16 で既定化)。Next の外では使わない
- **Rspack** — webpack 互換 API の Rust 実装。**巨大 webpack 資産の移行先**として堅調
- **esbuild** — 単体採用は減り、各ツールの内部部品として偏在

つまりバンドラは「選ぶもの」から「**フレームワークに付いてくるもの**」になりました。
直接選定する場面は減っています。

---

## 6. テスト — 決着済みだが「何をテストするか」は流派あり

- **Vitest** — ユニット/コンポーネントテストのデファクト(Jest からの世代交代は完了)
- **Playwright** — E2E のデファクト(Cypress は退潮)
- **Testing Library** — 「実装詳細でなく振る舞いをテストする」哲学ごと標準化
- **MSW** — API モックの標準

道具は決着しましたが、**テスト戦略**には流派が残ります。2026 年の多数派は
「**ユニットテストを大量に書くより、統合テスト(実際に近い形で動かす)に重心を**」
(いわゆる Testing Trophy)。ただし Go 文化圏はテーブル駆動ユニットテストの伝統が
強く([go-fable-101](../go-fable-101/README.md) で学んだ通り)、
言語コミュニティごとに重心が違うことは知っておく価値があります。

## 7. Python 側の一言(戦争終結のお知らせ)

Python のツールチェーン戦争は、Rust 製ツールによって**きれいに終わりました**。

- パッケージ/環境管理: **uv**(pip/venv/pyenv/Poetry の役割を一本化。Rust 製)
- リンタ/フォーマッタ: **Ruff**(flake8+isort+black を一本化。Rust 製)
- Web: **FastAPI + Pydantic** が API 開発の標準

JS より対立が少ないのは、後発だった分「JS の教訓を輸入して一気に統一」できたからです。

## 8. AI コーディング時代のツールチェーン

[01 章](01_principles.md)で述べた「AI に書かせやすさ」は、ツール選定にも直結します。

- **速いリンタ・型チェッカ・テストは「AI の暴走防止柵」**。AI がコードを量産する
  ほど、機械的な検証ループ(lint → typecheck → test)の速度が開発速度を律速する。
  Rust/Go 製ツールへの世代交代が加速した一因はこれ
- エディタ内蔵 AI(Copilot)→ 対話型(Cursor)→ **エージェント型(Claude Code 等、
  ツールを自分で実行して検証まで回す)** へと重心が移動中
- リポジトリに AI 向け説明書(CLAUDE.md / AGENTS.md)を置き、
  ドキュメントを llms.txt で公開するのが 2026 年の新しい作法
